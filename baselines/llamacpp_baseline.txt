================================================================================
KEY FINDINGS
================================================================================

1. OPTIMAL THREAD CONFIGURATION:
   - Prefill (batched):  8 threads → 25.29 t/s
   - Decode (sequential): 2 threads → 9.05 t/s
   - Strategy: Switch thread count between prefill and decode phases

2. THE NUMBERS TO BEAT:
   - Prefill (time-to-first-token for 512 tokens): 25.29 t/s → ~20.2 seconds
   - Decode (sustained generation): 9.05 t/s (best), 5.91 t/s (sustained 512 tokens @ 8t)
   - Target for QNN HTP runtime: >15 t/s decode, >40 t/s prefill

3. THERMAL BEHAVIOR:
   - Cold start: 32-35°C
   - After full benchmark suite: 42.4°C (peak)
   - No throttling observed (max temp well under 55°C limit)
   - All benchmarks ran at stock CPU governor settings (no performance mode)

4. MEMORY USAGE:
   - Model RSS: ~1.3 GB (measured via /proc/PID/status VmRSS)
   - System memory remained above 5.8 GB throughout (well above 1 GB floor)

5. WHY 4-THREAD BIG-CORE-ONLY IS SLOW:
   - The 4-thread taskset 0xF0 result (1.49 t/s decode) is an artifact of
     pinning to only big cores. The big cores are faster per-core but the
     decode phase is memory-bandwidth-bound, not compute-bound. Using all
     8 cores (including LITTLE cores) provides more memory bandwidth
     parallelism, yielding 3.5x better decode throughput.

   - The 2-thread result (9.05 t/s) is even better because OpenMP thread
     synchronization overhead dominates at these small batch sizes.
     2 threads minimizes sync cost while still utilizing enough bandwidth.
